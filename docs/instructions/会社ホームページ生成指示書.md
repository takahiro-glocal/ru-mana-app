# 会社ホームページ生成指示書 — るうまな（RUUMANA）

> **目的:** 本ドキュメントは、本アプリのデザイン仕様（`/DESIGN.md`）と各種ドキュメント（`/docs/`）を Source of Truth として、サービス紹介を兼ねた「会社（運営チーム）公式ホームページ」を AI コーディングエージェントに生成させるための指示書である。
>
> **想定される実行者:** Claude / GPT 等の LLM コーディングエージェント、または LP 制作担当の人間。
>
> **想定アウトプット:** 静的サイト（HTML / CSS / 軽量 JS、または Nuxt/Next の単一ページ）。デザインは本アプリと一貫し、サービスの世界観を訪問者に1ページで伝えられること。

---

## 1. 出力物の仕様

### 1.1 ファイル構成

以下のいずれかで出力する。プロジェクト側の指示がなければ **A** を採用する。

**A. 単一 HTML + Tailwind（推奨）**

```
website/
├── index.html              # 日本語版（既定）
├── en/index.html           # 英語版
├── zh/index.html           # 中国語版
├── assets/
│   ├── style.css           # Tailwind ビルド済み or カスタム
│   ├── logo.png            # /public/images/logo.png を流用
│   └── screenshots/        # アプリのスクリーンショット
└── README.md               # ビルド／公開手順
```

**B. Nuxt ページ追加（既存アプリに同居）**

`pages/company.vue` または `pages/about.vue` を作成し、本指示書の構成に従う。スタイルは既存の Tailwind 設定（`prefix: 'tw-'`）を使う。

### 1.2 技術的制約

- レスポンシブ必須（モバイル 375px 〜 デスクトップ 1440px）
- 画像は遅延読み込み（`loading="lazy"`）
- CSS は Tailwind（既存設定 `ru-shiru` / `ru-miru` / `ru-kiku` / `ru-iku` / `ru-trouble` / `ru-bg` を使用）
- JS は最小限（スクロール演出・モバイルメニュー程度）
- アクセシビリティ: WCAG AA、`prefers-reduced-motion` 尊重、フォーカスリング保持
- 多言語対応: `<html lang>` を正しく設定、3言語切替は右上ピル型ボタン

## 3. ページ構成（必須セクション）

順序通りに以下を構成すること。

### Section 1: ヒーロー

- 背景: 空のグラデーション（朝〜夕の青〜橙）または `#F9F5E7`
- 中央上: ロゴ（円形リング枠）
- サービス名「るうまな（RUUMANA）」
- タグライン: **「異文化の摩擦を、理解へ。」**
- 主要 CTA: 「アプリを使ってみる」（ピンク）／「資料をダウンロード」（白）
- 右上: 言語切替（JA / EN / ZH）

### Section 2: ミッション

- 見出し: **「違いを、楽しもう。」**
- 本文（既存コピーをそのまま使用）:
  > 言葉だけでなく、文化やマナーの違いによる誤解や不安を、事前に楽しく学べる環境を提供します。「否定しない」「争わない」——文化の違いを、出会いのきっかけに。

### Section 3: キーメッセージ 3点

## 4. コピー作成ルール

### 4.1 そのまま使うべき公式コピー

以下は `i18n/locales/ja.json` の `landing.*` キーに既にある公式コピー。**改変せずに引用** すること。

- `app_name`: 「るうまな」
- `tagline`: 「異文化の摩擦を、理解へ。」
- `concept_title`: 「違いを、楽しもう。」
- `concept_body`: 上記 Section 2 本文
- `features_title`: 「4つの『まな』で、文化をつなぐ」
- `feature_shiru` 〜 `feature_iku`: 各カードの説明文
- `cta_title`: 「さあ、はじめよう」
- `cta_subtitle`: 「ブラウザですぐ使える。アプリとしてインストールも。」

### 4.2 新規コピーを書く時

- 1文 40 字以内を目安
- 命令形は CTA のみ（「使ってみる」「相談する」など短く）
- 「外国人」を主語にしない。「旅する人」「日本で暮らす人」「あなた」などに置き換える
- 数値は出典付き

## 5. 多言語対応

3言語（JA / EN / ZH）を生成する。日本語版を起点とし、以下の方針で訳す。

- 既存の `i18n/locales/{en,zh}.json` に該当キーがある場合は **その訳語をそのまま採用**
- 新規コピーは **意訳** する（直訳禁止）。文化的ニュアンスを保つ
- 各言語版は別 URL（`/`, `/en`, `/zh`）に配置し、相互リンクを `<link rel="alternate" hreflang>` で明示

## 7. アクセシビリティ・性能

- Lighthouse スコア目標: Performance ≥ 90、Accessibility ≥ 95、SEO ≥ 95
- `meta description`、OG 画像、Twitter Card を全ページに設定
- 画像は WebP 推奨、`width`/`height` を明示してレイアウトシフトを防止
- フォーカスリング消去禁止、リンクは `:hover` / `:focus-visible` を区別
- カラーだけで意味を伝えるな（バッジにはテキスト併用）

## 8. 生成手順（エージェント向け）

1. 上記「必読資料」を順に読み、特に `DESIGN.md` の色・トーン・コンポーネント仕様を内部メモにまとめる
2. ファイル構成（§1.1）を決定し、雛形を作る
3. Section 1〜12 を順に実装。各セクションごとにモバイル → デスクトップで確認
4. 日本語版を完成させてから英語・中国語版を生成（既存 i18n に訳がない箇所は意訳）
5. Lighthouse でセルフチェックし、目標未達なら最適化（画像圧縮、CSS purge、不要 JS 削除）
6. 生成したファイル一覧と、未確定のプレースホルダ（会社名・SNS リンクなど）をプロジェクトオーナーに **明示的に報告** する

---

## 9. やってはいけないこと（NG リスト）

- DESIGN.md のカラーパレットを独自に変更する
- タグラインや看板コピーを意訳でも書き換える（既定言語の日本語は厳守）
- 「正しいマナー／間違ったマナー」と二項対立で語る
- ステレオタイプな日本表現（富士山・芸者・忍者など）を強調する
- 緊急機能（おたすけ）を「便利機能」として軽い扱いで掲載する
- 数値や事業計画を、出典を確認せずに脚色する
- ダミーの SNS URL / 連絡先メールアドレスを placeholder としてリンクさせる（テキストプレースホルダのみ可）

## 付録 B: 5理念（フッターか「私たちの考え方」セクションに掲載推奨）

1. 否定しない — 文化に優劣はない。違いを否定しない
2. 争わない — 文化の違いで対立しない。橋渡し役になる
3. 個を尊重する — 国籍ではなく、一人ひとりの「個」を大切に
4. 共生 — 相互理解に基づく共生社会の実現
5. 楽しく学ぶ — 堅苦しくない、温かく楽しい文化学習
